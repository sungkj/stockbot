# 키움증권 계좌 잔고 및 예수금 조회 기능 구현 계획 (Plan)

## 1. 개요
- `docs` 디렉토리에 있는 `kiwoom_api_가이드.txt`와 `키움 REST API 문서.xlsx` 문서를 바탕으로, 사용자의 계좌별 예수금 및 보유 종목 잔고 현황을 조회하여 텔레그램 봇으로 전송하는 기능을 구현합니다.
- 현재 문서의 내용과 관련없는 부분은 절대 수정하지 않습니다.

## 2. 구현 단계별 계획

### 1단계: API 문서 분석 및 환경 설정 
- `docs` 디렉토리의 문서를 리뷰하여 예수금 상세 조회(예: opw00001) 및 계좌 평가 잔고 내역(예: opw00018)에 해당하는 REST API 엔드포인트 파악.
- API 호출에 필요한 인증 방식 및 필수 파라미터(계좌번호, 비밀번호 등) 요구사항 정리.
- `config.yaml`에 키움 API 연동을 위한 설정(Base URL, 계좌번호 등) 항목 추가.
- `kiwoom_keys` 디렉토리에 위치한 계좌별 키 파일(예: `[계좌번호]_appkey.txt`, `[계좌번호]_secretkey.txt`)을 열어 계좌별 인증 정보(App Key, Secret Key)를 로드하는 로직 추가.
- 계좌번호 리스트는 `kiwoom_keys` 디렉토리의 파일을 기반으로 만듭니다.

### 2단계: API 호출 로직 작성 (`tasks/kiwoom_api.py`)
- `KiwoomApiTask` 클래스 내에 API 서버와 통신할 비동기 HTTP 요청 로직 추가.
- 키움 API의 래퍼(Wrapper) 클래스를 제작하여, 각 API명(예: "계좌번호조회", "예수금상세조회")마다 독립된 함수로 제공하도록 모듈화.
- 코드 실행 시 각 계좌별 인증 정보를 사용하여 **계좌별로 개별적인 접근 토큰(Access Token)을 발급**받고 딕셔너리 형태로 관리하는 로직 구현.
- 계좌별 예수금 조회 메서드 설계.
- 계좌별 잔고/종목 현황 조회 메서드 설계.
- 통신 에러 및 API 제한(Rate Limit)에 대응하기 위한 예외 처리 구조 마련.

### 3단계: 메시지 포맷팅 및 가공 로직
- 반환된 원본 데이터에서 핵심 정보(예수금, 총 매입금액, 총 평가손익, 수익률, 개별 종목명, 보유 수량 등)를 추출하는 파서(Parser) 구현.
- 텔레그램에서 가독성이 좋도록 이모지와 줄바꿈을 활용한 메시지 템플릿 작성. (예: 마크다운 또는 HTML 파싱 적용)

### 4단계: 스케줄링 및 봇 기능 연동 (`core/task_manager.py` / `core/bot_app.py`)
- 사용자가 `/잔고` 명령어를 입력했을 때 즉각적으로 API를 호출하여 응답하는 핸들러 추가 (`bot_app.py`).
- 정해진 시간(평일 장 마감 후 15:40)에 자동으로 계좌 요약을 발송하도록 스케줄러 등록 (`task_manager.py`).
- 매일 새벽 3시에 키움 API 접근 토큰을 **각 계좌별로 모두 자동으로 재발급**받도록 갱신 스케줄러 등록 (`task_manager.py`).

### 5단계: 테스트 및 안정화
- 통신 실패 상황(장 점검 시간, 네트워크 오류 등)에 대한 안전한 폴백(Fallback) 메시지 처리 테스트.
- 실제 API 엔드포인트에 붙여 정상적으로 메시지가 수신되는지 최종 검증 및 포맷 다듬기.
